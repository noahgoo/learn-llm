The embedding matrix **W<sub>E</sub> ∈ ℝ<sup>|V|×d<sub>model</sub></sup>**
is a plain lookup table: token id *i* selects row *i*. No computation —
just indexing. The rows are ordinary parameters, learned end-to-end by
gradient descent. Model "width" largely *is* d<sub>model</sub>: 768 for
the GPT-2 small running here,<Cite id="radford2019" /> tens of thousands
in the largest published models.

Many GPT-family models (including this one) **tie** W<sub>E</sub> with
the output head: the same matrix that embeds inputs is transposed to
project the final hidden state back onto vocabulary logits, keeping one
consistent token space end to end.<Cite id="radford2019" />

The 3-D cloud is a PCA projection of the live model's W<sub>E</sub> rows
— three principal components out of 768 — so distances are heavily
compressed, but neighborhood structure survives. Your tokens' actual
768-d embedding vectors (the `tok_emb` output of the model in your
browser) are projected with the same components.
